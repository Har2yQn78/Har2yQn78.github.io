Most chatbot demos are impressive because they'll answer anything. Pulse is built around the opposite instinct: it should refuse to answer the moment it isn't sure. That single constraint shaped almost every technical decision in the system, down to index types and cache invalidation, so this post goes deep on the mechanics — not a feature list, but how each layer actually works and why it exists.

Pulse is a multi-tenant SaaS, built at [ZimmerAI](https://zimmerai.com), that runs document-grounded knowledge bots over Telegram and a web widget. Each tenant is one subscribed customer, brings their own LLM key, and only ever sees their own bots, documents, and conversations. Here's the shape of it, simplified from the real system diagram:

## Multi-tenant, but not by convention

The dashed boundary in that diagram is doing the real work, and it's worth being precise about what it actually is. It is not an app-level `WHERE tenant_id = ?` sprinkled through every query in the codebase, trusting every engineer to remember it on every new endpoint forever. It's Postgres row-level security: a policy attached to the table itself, keyed to a session variable the connection sets at the start of a request. Even a query that forgot the tenant filter entirely still can't see another tenant's rows, because the database refuses to hand them over — the enforcement point moved from "every call site, forever" to "one policy, once."

Identity and billing live entirely outside this system. A tenant logs in through Zimmer's marketplace panel, which is separate infrastructure Pulse talks to over the public internet like any external API. Pulse holds no local password, ever. The login flow is a redirect dance: hit Pulse with no session, get bounced to Zimmer's SSO authorize page, log in there, get redirected back with a short-lived `sso_token`, and Pulse verifies that token server-to-server against Zimmer's API using a shared marketplace secret before it ever sets a session cookie. Subscription state isn't polled either — Zimmer pushes `subscription.activated` / `subscription.revoked` webhooks, signed with that same secret, so a cancelled subscription cuts off access within one webhook round trip instead of waiting for the next eligibility check.

The other tenant-facing decision is BYOK — bring your own key. Every bot's generation and embedding calls run on the tenant's own OpenAI or Anthropic credential, not a pooled account Pulse owns. Those credentials are encrypted at rest with Fernet (symmetric, authenticated encryption), derived from a single server-side secret, so a database dump alone doesn't leak usable API keys. BYOK isn't primarily a cost-saving trick. It's a trust boundary: the tenant's documents are never processed by a model account someone else controls, and revoking access is rotating one key, not opening a support ticket and waiting.

## What a RAG system actually is, and where it breaks

Retrieval-Augmented Generation, in the simplest version, is three steps: turn the question into a vector, find document chunks with similar vectors, hand those chunks to an LLM and ask it to answer using only them. That version is easy to build, works fine in a demo with ten documents, and degrades badly in production. Almost none of the degradation is the LLM's fault. It's the retrieval and the data feeding it.

**Embedding and indexing.** Chunks are embedded with OpenAI's `text-embedding-3-large`, truncated to 1536 dimensions rather than the model's full native width — enough semantic resolution for retrieval, at a smaller storage and index cost per chunk. Those vectors sit in a pgvector column with an HNSW index for approximate nearest-neighbor cosine search, which is what makes similarity search stay fast as a knowledge base grows into the tens of thousands of chunks instead of degrading linearly with a flat scan.

**Why vectors alone aren't enough.** Pure vector similarity is a semantic instrument, and semantic instruments are bad at exact tokens. A user asking for "invoice #4471" or a specific error code wants the chunk containing that literal string, not the chunk that's merely in the neighborhood of "invoice" in embedding space — those can be different chunks entirely. So every chunk also gets a generated `tsvector` column, Postgres-native full-text search, backed by a GIN index, computed automatically from the chunk content so there's no separate sync job to keep it current. Retrieval runs both searches — vector cosine and full-text — over the same table in parallel, then fuses the two ranked result lists with Reciprocal Rank Fusion: each chunk's final score is the sum, across every list it appears in, of one over its rank in that list plus a constant. A chunk that ranks highly on either signal, semantic or lexical, surfaces near the top of the fused list, and a chunk that only one method found still gets credit instead of being drowned out by the other method's scores.

**Multi-turn and cross-lingual.** Retrieval assumes the thing you're embedding is a self-contained question, and real conversations are almost never that. "What about the enterprise tier?" carries no retrievable meaning as a vector unless you already know what was asked two turns earlier. Before anything gets embedded, Pulse rewrites the raw message into a standalone query — folding in enough conversation history to stand alone — using a cheap model call on the bot's own credential. The same rewrite step translates the query into the knowledge base's detected language, so a Farsi-speaking user can ask a question against an English-only manual and retrieval still lands on the right chunks; the model then replies back in whichever language the user actually asked in, not the KB's language. This step is a flag (`QUERY_REWRITE_ENABLED`), because it's one extra cheap LLM call per turn and some deployments would rather skip the cost for simple, single-turn, single-language bots.

**Semantic cache.** Before any of the above runs on the expensive path, the standalone query is checked against a semantic answer cache — a previous query embedding within cosine similarity 0.95 or higher returns the cached answer immediately, skipping retrieval and generation entirely. The threshold is deliberately tight: 0.95 is close enough to catch genuine near-duplicate phrasing of the same question, not loose enough to return a cached answer for a merely related one. That cache gets invalidated the moment new documents finish ingesting into that bot's knowledge base, because a correctly-grounded answer from before an upload can become an incorrect one after it, and a stale cache hit would silently keep serving the old answer with no error to notice.

**Dedup and rerank.** A cache miss runs the hybrid search and pulls a candidate pool — 20 chunks by default (`RETRIEVAL_CANDIDATE_K`). Those 20 often contain near-duplicates: the same paragraph appearing once in the main document and again in an appendix, or three chunks all restating the same sentence with slightly different surrounding context. Before reranking, Pulse dedups that pool with Maximal Marginal Relevance, which greedily selects chunks that are both relevant to the query and dissimilar to what's already been picked — so the five slots that make it to the model are five distinct pieces of information, not the same fact five times crowding out something that actually would have helped. What survives dedup goes through a listwise LLM rerank: the model sees the full candidate set at once and ranks it by actual relevance to the question, rather than trusting the retrieval similarity score to have already sorted it correctly. Similarity is a proxy for relevance, computed cheaply so it can scale to thousands of chunks; reranking is more expensive and only runs on the shortlist, replacing the proxy with the judgment it was standing in for. The survivors — five by default (`RERANK_TOP_K`) — are what actually reaches the generation prompt.

## Smart chunking: the part nobody demos

Every layer above is wasted effort if the chunks themselves are bad, and chunking is the least glamorous part of a RAG pipeline and the one most likely to quietly wreck answer quality without ever throwing an error.

**Extraction.** Text comes out of PDF, TXT, MD, and DOCX files through a per-type extractor. PDFs go through `pypdf` first; if a page comes back with no extractable text — a scanned image, not real text layer — it falls back to OCR, with language models for English, Farsi, and Arabic, so a scanned Farsi contract doesn't silently produce an empty chunk. Before any extractor runs at all, the upload is validated by magic bytes: actual content sniffing via libmagic, not the filename extension and not the client-supplied MIME type. A `.exe` renamed to `.pdf` gets rejected before it reaches a parser, and the detected type is what selects the extractor, not what the client claimed it was.

**Chunking.** The naive approach — cut every 500 characters, done — is fast and wrong. It slices sentences in half, separates a table from its header row, and severs a claim from the qualifying clause three words later. A chunk that ends mid-sentence hands the retrieval-and-generation pipeline half a fact, and the model will confidently generate the wrong other half because nothing told it the sentence was truncated. Pulse chunks recursively instead: split on paragraph boundaries first, and only fall back to splitting a paragraph on sentence boundaries if it's too large to fit as one chunk, and only fall back further to word boundaries if even a single sentence is oversized. Sentence boundaries are detected with `pysbd`, a real sentence-boundary library, instead of a naive split on periods — which matters more than it sounds, because "Dr. Smith approved the Q3 budget." is one sentence, not two, and a period-splitter will cut it there anyway, right in the middle of a name. The result is chunks that end where a human reading the document would naturally end them: complete thoughts, not fragments.

**After ingestion finishes.** Storing chunks isn't the last step. The pipeline also detects the knowledge base's dominant language from the ingested content — the language the cross-lingual query rewrite translates into — and generates a first draft of FAQs from the material, which an operator can curate rather than write from scratch. Then, as noted above, it invalidates the semantic answer cache for that bot, so the next matching query is guaranteed to see the newly ingested content instead of a cached answer from before it existed.

## The guardrail: two chances to say "I don't know"

This is the section the rest of the pipeline exists to support, so it's worth being exact about the mechanism, because "grounding" as a word gets used loosely enough to mean almost nothing.

Grounding is not a prompt instruction. "Please only use the provided context" is a request a model can and does ignore under enough ambiguity or a sufficiently leading question — it's a preference stated in natural language, not an enforced constraint. Pulse enforces it structurally, as two separate, independently-failing gates around the generation call.

**Gate one, pre-generation.** If reranking comes back and nothing in the survivors is actually relevant to the question — a real "no chunk cleared the bar" case, not just a low top score — Pulse never calls the generation model at all. It returns a fallback message directly. No relevant evidence in the knowledge base means no attempt to answer from thin air, and the tenant's LLM credential doesn't get charged for a call the system already knows can't be grounded.

**Gate two, post-generation.** When there is relevant context, generation runs, but not as free text — the model is asked for structured output: the answer itself, the specific passage indices it's citing, and a `grounded` boolean the model sets as a self-report on whether its own answer is actually supported by what it was given. If `grounded` comes back false, Pulse discards the answer and returns the same fallback path as gate one, even though generation already ran. The cited passage indices aren't cosmetic — they're mapped back to real source document filenames and surfaced to the user alongside the answer, so grounding is a checkable claim (here's exactly where this came from) rather than an assertion (trust me).

**Feedback loop.** Every Nth answer — configurable, five by default (`FEEDBACK_EVERY_N`) — prompts the user for a 👍 or 👎, which is logged to `query_logs` alongside the query and the chunks that were used. That's not just a UX nicety; it's the raw material the eval harness pulls from, covered below.

## Human handoff: the same instinct, a different threshold

Grounding gates catch the case where the system can detect its own uncertainty. They don't catch everything — a question can be answerable-sounding, clear a relevance bar, generate a `grounded: true` answer, and still be wrong, or a user can simply want a human regardless of how good the bot's answer was. Handoff exists for both cases.

Two triggers route a conversation to a human: the message matches a configured escalation keyword, or the conversation is already flagged low-confidence by the gates above. Once escalated, a conversation flips into a human or awaiting-human state, and the state machine that gates every inbound message checks that first, before retrieval ever runs — a conversation already in human mode gets published straight to the operator inbox with no bot reply generated at all, so an operator and the bot are never both answering the same thread. The inbox updates in realtime over Redis pub/sub, including a typing indicator forwarded from the widget, so the handoff feels like a live handoff rather than a form dropped into a queue. It's the same underlying instinct as the grounding gates, just at a different confidence threshold: gate one and two say "I don't have evidence, so I won't answer"; handoff says "I'm not confident enough here even with evidence, so a person should".

## Eval: start small, grow it from real failures

None of the above is verifiable without measurement, and "the bot passed its own grounding check" is not evidence the grounding check is well-calibrated — it's exactly the kind of self-referential claim the two gates exist to prevent everywhere else in the system. So Pulse has a separate, offline eval harness that doesn't trust the live gates at all.

It runs the RAGAS metric set against a per-bot golden set of question/expected-answer pairs, stored as one JSON file per bot: **faithfulness** (is the generated answer actually supported by the chunks that were retrieved, checked independently of the model's own `grounded` self-report), **answer relevancy** (does the answer address what was actually asked, not just something adjacent to it), **context precision** (of the chunks retrieved, how many were actually relevant — a noisy retriever that happens to generate fine answers still scores badly here), and **context recall** (of the chunks in the knowledge base that were actually relevant to the question, how many did retrieval surface — the metric that catches a retriever that's too narrow, not just noisy). Crucially, eval runs execute the real production pipeline, not a mocked one, but read-only: no writes to the answer cache, no writes to `query_logs`. An eval run has to exercise the exact code path a real user hits, or it isn't measuring the thing that matters, but it can't be allowed to pollute production analytics or seed the cache with eval-only queries, or the two purposes bleed into each other.

The deliberate choice is to not start with a large hand-authored golden set. A big upfront eval set mostly encodes the questions someone thought to ask before launch, which is a biased sample of the questions real users actually break the bot with. Instead, the golden set starts small and grows from production signal: every 👎 an operator sees in the inbox gets pulled into the golden set as a new eval case by a seeding script, with the operator filling in the one missing piece — the correct answer — once. Run the eval, get a baseline. Change something in retrieval. Run it again, compare. The suite is a running record of actual failures, not a guess at hypothetical ones, which means it gets more useful the longer the bot has been in front of real users, instead of stalling out as a snapshot of launch-week assumptions.

## Why the constraint is the product

None of these pieces — row-level security enforced at the database, BYOK credentials, hybrid retrieval fused with RRF, MMR dedup, listwise rerank, sentence-safe recursive chunking, a semantic cache that invalidates itself on new content, two independently-failing groundedness gates, human handoff at a second confidence threshold, an eval suite seeded from real thumbs-down — is exotic in isolation. Any one of them is a reasonable thing to find in a well-built retrieval system. What makes them add up to something coherent is that every one of them is in service of the same refusal: don't answer past what the evidence supports, and make that refusal checkable at every layer instead of promised once at the prompt. The interesting engineering problem in a knowledge bot was never getting it to talk. It was building enough structure, at enough layers, that it reliably knows when to stop.
